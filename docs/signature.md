# ฟีเจอร์ลายเซ็นผู้ใช้ (Saved Signature)

เอกสารนี้อธิบายฟีเจอร์ **ลายเซ็นผู้ใช้** ในระบบจองพื้นที่ (Asset/Space Booking System) —
ให้ผู้ใช้วาดลายเซ็นด้วยมือหรืออัปโหลดไฟล์ PNG พื้นหลังโปร่งใส เพื่อแนบไปกับฟอร์มขอใช้พื้นที่
(booking) โดยไม่ต้องเซ็นเอกสารกระดาษ และถ้าต้องการก็บันทึกลายเซ็นนั้นไว้ในระบบเพื่อใช้ซ้ำ
ในครั้งถัดไปได้เลยโดยไม่ต้องวาดใหม่ทุกครั้ง

เอกสารนี้เขียนให้อ่านจบแล้วเข้าใจฟีเจอร์นี้ได้ครบ โดยไม่ต้องเปิดไฟล์อื่นประกอบ

> อัปเดตล่าสุด: 2026-07-15

---

## 1. ภาพรวมของระบบ

ระบบนี้เขียนด้วย **Go** (backend, ให้บริการ REST API) และ **Next.js/TypeScript** (frontend)
ไฟล์ (รูปภาพ/เอกสาร) ทุกชนิดในระบบ — รวมถึงลายเซ็น — ไม่ได้เก็บไว้ในฐานข้อมูล SQL โดยตรง
แต่เก็บไว้ใน **MinIO** ซึ่งเป็นระบบเก็บไฟล์แบบ object storage (พูดง่ายๆ คือคล้าย Amazon S3
แต่รันเองได้บนเครื่องของทีม) ฐานข้อมูล SQL เก็บแค่ "ที่อยู่ของไฟล์" (เรียกว่า `object_key`)
ไว้ตามหาไฟล์จริงทีหลัง

เวลาจะให้ผู้ใช้ดาวน์โหลด/ดูไฟล์ ระบบจะสร้าง **presigned URL** — ลิงก์ชั่วคราวที่ใช้ได้ในเวลาจำกัด
(เช่น 15 นาที) แทนที่จะเปิดให้เข้าถึงไฟล์ได้แบบสาธารณะตลอดไป พอ URL หมดอายุ ต้องขอสร้างใหม่

ฟีเจอร์ลายเซ็นใช้กลไกนี้เหมือนกัน:

```
                    ┌───────────────────────────┐
SignaturePad.tsx ──▶│  POST /api/v1/me/signature │──▶ ตรวจสอบไฟล์ (4 ขั้นตอน)
(วาด/อัปโหลด)        │  (บันทึกลายเซ็นไว้ใช้ซ้ำ)     │      │
                    └───────────────────────────┘      ▼
                                                  เก็บไฟล์ลง MinIO
                                                  โฟลเดอร์: "signatures"
```

ลายเซ็นในระบบนี้ถูกใช้งาน **2 ทางที่แยกจากกัน**:

1. **แนบกับ booking โดยตรง (ครั้งเดียว)** — ผู้ใช้วาด/อัปโหลดลายเซ็นแล้วส่งไปพร้อมฟอร์ม
   เอกสารขอใช้พื้นที่ทันที (ผ่านคอมโพเนนต์ `DocumentFormModal.tsx`) กรณีนี้ **ไม่ผ่าน API
   `/me/signature` เลย** — ลายเซ็นเป็นแค่รูปภาพ (data URL) ที่แปะลงในเอกสาร/PDF ที่กำลังสร้าง
   ไม่ได้ถูกบันทึกไว้ในระบบถาวร
2. **บันทึกไว้ใช้ซ้ำ (ผ่าน API `/me/signature`)** — ลายเซ็น "ของฉัน" ที่บันทึกไว้ครั้งเดียว
   ครั้งต่อๆ ไปกดเลือกใช้ได้ทันทีโดยไม่ต้องวาด/อัปโหลดใหม่ จัดการได้จาก 2 จุด: หน้าโปรไฟล์ผู้ใช้
   (`ProfileView.tsx`) และตอนกรอกฟอร์มจองห้อง (`SignaturePad.tsx` โหมด "ใช้ลายเซ็นที่บันทึกไว้")

ผู้ใช้ 1 คนมีลายเซ็นที่บันทึกไว้แบบนี้ได้ **สูงสุด 1 อัน** — ถ้าบันทึกใหม่ อันเก่าจะถูกแทนที่
(ลบไฟล์เก่าออกจากระบบเก็บไฟล์จริงด้วย ไม่ใช่แค่ในฐานข้อมูล)

## 3. Backend API: `/api/v1/me/signature`

ทุก endpoint ในกลุ่มนี้**ต้องล็อกอินก่อน** (ส่ง JWT token มาด้วย) และทำงานกับลายเซ็น
ของผู้ใช้ที่ล็อกอินอยู่เท่านั้น — ไม่มีทางระบุ user คนอื่นผ่าน parameter ได้

| Method | ใช้เมื่อ |
|--------|----------|
| `POST` | บันทึกลายเซ็นใหม่ (ถ้ามีอันเก่าอยู่แล้ว จะถูกแทนที่) |
| `GET`  | ดึงลายเซ็นที่บันทึกไว้ (จะได้ presigned URL ใหม่ทุกครั้งที่เรียก เพราะ URL เก่าอาจหมดอายุแล้ว) |
| `DELETE` | ลบลายเซ็นที่บันทึกไว้ทิ้ง |

### `POST` — บันทึกลายเซ็น

**Request** ส่งแบบ `multipart/form-data`:

| field | ชนิด | บังคับ |
|-------|------|--------|
| `file` | ไฟล์ (ต้องเป็น PNG เท่านั้น) | ✅ |

**Response สำเร็จ (HTTP 201)**
```json
{
  "success": true,
  "data": {
    "url": "http://localhost:9000/asset-sut-bucket/signatures/20260715...png?X-Amz-...",
    "updated_at": "2026-07-15T10:00:00+07:00"
  }
}
```

**กรณีผิดพลาด (HTTP 400)**

| สาเหตุ | ข้อความที่ตอบกลับ |
|--------|---------------------|
| ไม่ได้แนบไฟล์มา | `missing form-data file field 'file'` |
| ไฟล์ใหญ่เกิน 1MB | `file too large (max 1MB)` |
| ไม่ใช่ไฟล์ PNG จริง (ตรวจจากเนื้อไฟล์ ไม่ใช่แค่นามสกุล) | `signature must be a genuine PNG file` |
| ขนาดภาพ (กว้าง/สูง) เกิน 4096x4096 พิกเซล | `signature PNG dimensions are too large` |
| ไฟล์เสียหาย เปิดเป็นภาพไม่ได้ | `could not decode PNG image` |
| พื้นหลังของภาพไม่โปร่งใสมากพอ | `signature PNG must have a transparent background` |

### `GET` — ดึงลายเซ็นที่บันทึกไว้

รูปแบบ response เหมือน `POST` ทุกประการ — แต่ถ้าผู้ใช้คนนี้ยังไม่เคยบันทึกลายเซ็นเลย
จะได้ **HTTP 404** กลับมาแทน (ถือเป็นกรณีปกติ ไม่ใช่ข้อผิดพลาด)

### `DELETE` — ลบลายเซ็น

**Response สำเร็จ (HTTP 200)**
```json
{ "success": true, "data": { "deleted": true } }
```

## 4. ขั้นตอนตรวจสอบไฟล์ก่อนบันทึก (4 ชั้น)

จุดสำคัญที่สุดของฟีเจอร์นี้คือ **การตรวจสอบไฟล์ที่ผู้ใช้อัปโหลด** เพราะรับไฟล์จากผู้ใช้ตรงๆ
จึงต้องมั่นใจว่าเป็น PNG พื้นหลังโปร่งใสจริง ไม่ใช่ภาพปลอมหรือภาพที่จะทำให้ระบบมีปัญหา

**หลักการสำคัญ: การตรวจสอบทุกขั้นตอนทำที่ฝั่ง backend (server) เสมอ** เพราะข้อมูลที่ browser
ส่งมา (เช่น header ที่บอกว่า "นี่คือไฟล์ชนิด PNG") ไม่น่าเชื่อถือ — ใครก็ตามที่เรียก API ตรงๆ
โดยไม่ผ่านหน้าเว็บ (เช่นเขียนสคริปต์ยิง API เอง) สามารถตั้งค่าพวกนี้เป็นอะไรก็ได้ตามใจ
ดังนั้นฝั่ง frontend ที่ตรวจสอบไฟล์ก่อนอัปโหลดนั้น **เป็นแค่การช่วยเตือนผู้ใช้ให้เร็วขึ้น
(ประสบการณ์ใช้งานที่ดีกว่า)** ไม่ใช่ตัวป้องกันจริง — ตัวป้องกันจริงอยู่ที่ backend เท่านั้น

backend ตรวจสอบไฟล์ตามลำดับนี้:

1. **ตรวจ Magic Bytes** — ไฟล์ PNG ทุกไฟล์จะขึ้นต้นด้วยชุดไบต์คงที่ 8 ไบต์เสมอ
   (`\x89PNG\r\n\x1a\n`) ระบบเช็คไบต์แรกของไฟล์จริงว่าตรงกับนี้หรือไม่ — วิธีนี้แม่นยำกว่า
   การดูแค่นามสกุลไฟล์ (`.png`) มาก เพราะเปลี่ยนนามสกุลไฟล์ได้ง่ายๆ แต่ปลอมเนื้อไฟล์จริงยากกว่า

2. **ตรวจขนาดไฟล์ (ตอนยังบีบอัดอยู่)** — ต้องไม่เกิน 1MB ตรวจตั้งแต่ก่อนแตกไฟล์ (decode)
   เพื่อไม่ให้เสียเวลาประมวลผลไฟล์ที่ใหญ่เกินไปโดยไม่จำเป็น

3. **ตรวจขนาดภาพจริง (หลัง decode)** — ต้องไม่เกิน 4096x4096 พิกเซล ตรวจโดยการอ่านแค่
   "หัวไฟล์" (header) ของ PNG ก่อน โดยยังไม่ต้องแตกไฟล์เต็มรูป เหตุผลที่ต้องเช็คขั้นนี้แยกจาก
   ขั้นที่ 2: ไฟล์ PNG ขนาดเล็ก (ผ่านเช็คขนาด 1MB สบายๆ) แต่บีบอัดเก่งมากๆ สามารถขยายตัว
   ตอน decode กลายเป็นภาพที่มีพิกเซลมหาศาลได้ (เรียกว่า "decompression bomb") ซึ่งจะทำให้
   เครื่อง server เสียเวลา/หน่วยความจำมหาศาลในการประมวลผลภาพนั้นทั้งที่ไฟล์เล็กนิดเดียว —
   การเช็คขนาดภาพก่อนจะแตกไฟล์เต็มรูปช่วยตัดปัญหานี้ตั้งแต่ต้น

4. **ตรวจว่าพื้นหลังโปร่งใสจริง** — ต้องมีอย่างน้อย **20% ของพิกเซลทั้งภาพ** ที่มีความโปร่งใส
   อย่างมีนัยสำคัญ (alpha ต่ำกว่าครึ่งหนึ่งของค่าสูงสุด) ไม่ใช่แค่โปร่งใสนิดเดียว

   > **ทำไมต้องเช็คแบบนี้แทนที่จะเช็คแค่ "มีพิกเซลโปร่งใสอย่างน้อย 1 จุด":**
   > ก่อนหน้านี้ระบบเช็คแค่ว่ามีพิกเซลโปร่งใสอย่างน้อย 1 จุดในภาพทั้งหมด ซึ่งเป็นช่องโหว่ —
   > ใครก็ตามที่มีภาพพื้นทึบทั้งภาพ (ไม่โปร่งใสเลย) สามารถแก้ไขแค่พิกเซลมุมเดียวให้มีความโปร่งใส
   > นิดเดียว (เช่น alpha=254 จาก 255) ก็จะ "หลอก" ผ่านการตรวจสอบได้ ทั้งที่ภาพนั้นไม่ได้มี
   > พื้นหลังโปร่งใสจริงตามเจตนา ระบบจึงเปลี่ยนมาเช็ค **สัดส่วน** ของพิกเซลโปร่งใสแทน เพราะ
   > ลายเซ็นจริงที่วาดจาก signature pad จะมีลักษณะเป็นพื้นโปร่งใสเกือบทั้งภาพ มีแค่เส้นปากกา
   > ทึบบางๆ เท่านั้น จึงผ่านเกณฑ์ 20% ได้อย่างสบายๆ ในขณะที่ภาพปลอมพื้นทึบ+พิกเซลเดียว
   > จะไม่ผ่านเกณฑ์นี้ (มี test ยืนยันพฤติกรรมนี้ไว้ใน `signature_test.go`)

ฝั่ง frontend (`SignaturePad.tsx`) มีโค้ดตรวจสอบคู่ขนานกันแบบเดียวกัน (ยกเว้นขั้นตอนที่ 2 และ 3
ที่ frontend ใช้เกณฑ์ขนาดไฟล์ของตัวเอง — ดูหัวข้อ 6) เพื่อเตือนผู้ใช้ก่อนที่จะกดอัปโหลดจริง
แต่การเช็คนี้ใช้เฉพาะตอนผู้ใช้เลือกโหมด "อัปโหลดไฟล์" เท่านั้น — โหมด "วาดเอง" ด้วยเมาส์/นิ้ว
ไม่ต้องตรวจสอบ เพราะพื้นหลังของ canvas ที่ใช้วาดนั้นโปร่งใสอยู่แล้วโดยธรรมชาติของมันเอง

## 5. การเก็บไฟล์จริง (Object Storage)

- ไฟล์ลายเซ็นถูกเก็บไว้ใน MinIO ผ่านตัวกลางชื่อ `StorageService` ในโฟลเดอร์ที่ตั้งชื่อคงที่ว่า
  `"signatures"` (แยกจากไฟล์ประเภทอื่น เช่น สลิปโอนเงิน ที่เก็บคนละโฟลเดอร์)

- ทุกไฟล์ที่อัปโหลดเข้าระบบ (ไม่ใช่แค่ลายเซ็น) จะถูกติด "ป้ายกำกับ" (metadata) ไว้ว่า
  `Uploaded-By: <รหัสผู้ใช้>` ตั้งแต่ตอนอัปโหลด กลไกนี้ใช้ป้องกันไม่ให้คนอื่นเอา "ที่อยู่ไฟล์"
  (object key) ของคนหนึ่งไปผูกเข้ากับ record ของอีกคนหนึ่งได้ (เช่น ถ้ามีคนแอบรู้ที่อยู่ไฟล์คนอื่น
  จากลิงก์ที่หลุดออกมา ก็จะเอาไปผูกกับบัญชีตัวเองไม่ได้ เพราะระบบเช็คป้ายกำกับนี้ก่อนเสมอ)

- ตาราง `user_signatures` ในฐานข้อมูลเก็บแค่ "ที่อยู่ไฟล์" (`object_key`) และชื่อที่เก็บ
  (`bucket_name`) เท่านั้น **ไม่ได้เก็บลิงก์ URL ถาวรไว้** เพราะ presigned URL หมดอายุได้
  ทุกครั้งที่มีการเรียก `GET /me/signature` ระบบจะสร้างลิงก์ใหม่ให้สดใหม่เสมอ

- เวลาบันทึกลายเซ็นใหม่ทับของเก่า (`POST` ครั้งที่ 2 ขึ้นไป) ระบบจะ**ลบไฟล์เก่าออกจาก
  ที่เก็บไฟล์จริงด้วย** ไม่ใช่แค่ลบข้อมูลอ้างอิงในฐานข้อมูลเท่านั้น เพื่อไม่ให้มีไฟล์ขยะค้างอยู่

## 6. ข้อสังเกต: ขนาดไฟล์สูงสุดของ Frontend กับ Backend ไม่เท่ากัน

| | Frontend | Backend |
|---|---|---|
| ขนาดขั้นต่ำ | 2 KB | ไม่มีการจำกัด |
| ขนาดขั้นสูง | 500 KB | 1,024 KB (1 MB) |

ความแตกต่างนี้**เป็นการออกแบบที่ตั้งใจ ไม่ใช่ข้อผิดพลาด** เหตุผลคือ backend อนุญาตขนาดไฟล์
ที่ใหญ่กว่า frontend เสมอ (500KB น้อยกว่า 1024KB) ทำให้ไฟล์ทุกไฟล์ที่ผ่านการตรวจสอบจาก
หน้าเว็บ (frontend) แล้ว จะไม่มีทางถูก backend ปฏิเสธซ้ำอีกเพราะเรื่องขนาด — ถ้ากลับด้านกัน
(backend เข้มงวดกว่า frontend) นั่นถึงจะเป็นปัญหาจริง เพราะผู้ใช้จะอัปโหลดผ่านหน้าเว็บได้ปกติ
แต่ไปเจอ error ตอนส่งข้อมูลจริงไปหา backend

ส่วนขนาดขั้นต่ำ 2KB ที่ frontend ตั้งไว้นั้น เป็นแค่คำเตือนเรื่อง**คุณภาพของภาพ** (ไฟล์เล็กเกินไป
อาจเป็นภาพที่ไม่คมชัด) ไม่ใช่มาตรการด้านความปลอดภัย ส่วนมาตรการป้องกันที่แท้จริง (การตรวจสอบ
4 ขั้นตอนในหัวข้อ 4) ไม่ได้พึ่งพาขนาดไฟล์เป็นเกณฑ์เลยแม้แต่น้อย จึงไม่มีความจำเป็นต้องบังคับ
ขั้นต่ำที่ backend ด้วย

หมายความว่า: ถ้ามีใครเรียก API ตรงๆ โดยข้ามหน้าเว็บไป จะสามารถอัปโหลดไฟล์ขนาดเล็กมากๆ
หรือใหญ่ถึง 1MB ได้ (กว้างกว่าที่หน้าเว็บอนุญาต) แต่ยังคงต้องผ่านการตรวจสอบ 4 ขั้นตอนใน
หัวข้อ 4 อยู่ดี จึงยังไม่ถือเป็นช่องโหว่ด้านความปลอดภัย

## 7. การทำงานฝั่ง Frontend: 3 โหมดของ `SignaturePad`

คอมโพเนนต์ `SignaturePad` มี 3 โหมดให้ผู้ใช้เลือก สลับไปมาได้ตลอดเวลา:

1. **โหมดวาดเอง (`draw`)** — ผู้ใช้วาดลายเซ็นด้วยเมาส์หรือนิ้ว (บนมือถือ/แท็บเล็ต) ลงบน
   พื้นที่วาดภาพ (canvas) ในเบราว์เซอร์โดยตรง ภาพที่ได้จะมีพื้นหลังโปร่งใสอยู่แล้วตามธรรมชาติ
   ของ canvas จึงไม่ต้องผ่านการตรวจสอบใดๆ เพิ่มเติม

2. **โหมดอัปโหลดไฟล์ (`upload`)** — ผู้ใช้เลือกไฟล์ PNG จากเครื่องตัวเอง (เช่น ลายเซ็นที่สแกน
   หรือถ่ายรูปมาแล้วตัดพื้นหลังออก) ไฟล์นี้จะผ่านการตรวจสอบ 4 ขั้นตอนตามหัวข้อ 4 (เวอร์ชัน
   ฝั่ง frontend) ก่อนที่จะยอมให้ใช้งานต่อ

3. **โหมดใช้ลายเซ็นที่บันทึกไว้ (`saved`)** — ถ้าผู้ใช้เคยบันทึกลายเซ็นไว้ในระบบมาก่อน
   (ผ่าน API `/me/signature`) ระบบจะดึงมาให้เลือกใช้ได้ทันที โดยจะพยายามโหลดข้อมูลนี้
   อัตโนมัติทันทีที่เปิดหน้าจอขึ้นมา

ในโหมด "วาดเอง" และ "อัปโหลดไฟล์" จะมีปุ่ม **"บันทึกไว้ใช้ครั้งถัดไป"** ให้กด ถ้ากด ระบบจะแปลง
ภาพที่วาด/เลือกไว้เป็นไฟล์แล้วส่งไปบันทึกผ่าน `POST /me/signature` ทันที

**การจัดการข้อผิดพลาดตอนโหลดลายเซ็นที่บันทึกไว้:** ระบบแยกแยะ 2 กรณีนี้อย่างชัดเจน —
(ก) กรณี "ยังไม่เคยบันทึกลายเซ็นเลย" (backend ตอบ 404) ถือเป็นเรื่องปกติ ไม่แสดงข้อผิดพลาด
ใดๆ ให้ผู้ใช้เห็น กับ (ข) กรณี "โหลดล้มเหลวจริงๆ" (เช่น อินเทอร์เน็ตหลุด, server มีปัญหา)
ซึ่งจะแจ้งเตือนผู้ใช้ให้ทราบ ไม่ปล่อยให้เข้าใจผิดไปว่า "ไม่มีลายเซ็นบันทึกไว้" ทั้งที่จริงๆ
แล้วมีอยู่แต่แค่โหลดไม่สำเร็จ

## 8. สรุปความพร้อมใช้งาน

### สิ่งที่ทำไว้ครบแล้ว

- ตรวจสอบไฟล์ครบ 4 ชั้น (รูปแบบไฟล์จริง, ขนาดไฟล์, ขนาดภาพ, สัดส่วนความโปร่งใส) ที่ backend ทั้งหมด
- ป้องกันปัญหา decompression bomb ด้วยการตรวจสอบขนาดภาพก่อนแตกไฟล์เต็มรูป
- ผูกไฟล์แต่ละไฟล์เข้ากับผู้อัปโหลดจริงผ่านป้ายกำกับ (metadata) ในระบบเก็บไฟล์ ป้องกันการ
  แอบเอาที่อยู่ไฟล์คนอื่นมาผูกกับบัญชีตัวเอง
- แยกกรณี "ไม่มีลายเซ็นบันทึกไว้" กับ "โหลดข้อมูลล้มเหลว" ออกจากกันอย่างชัดเจนทั้งฝั่งหน้าบ้าน

### สิ่งที่ยังไม่ได้ทำ / น่าพิจารณาเพิ่มเติมในอนาคต

- backend ยังไม่มีการจำกัดขนาดไฟล์ขั้นต่ำ (เป็นการตัดสินใจที่ตั้งใจไว้ตามหัวข้อ 6 แต่ถ้าต้องการ
  เพิ่มการป้องกันอีกชั้นก็ปรับเพิ่มได้ภายหลัง)
- ยังไม่มีการจำกัดความถี่ (rate limit) เฉพาะสำหรับ endpoint นี้ — ผู้ใช้สามารถยิงคำขอบันทึก
  ลายเซ็นซ้ำๆ ติดต่อกันได้ (แต่ละครั้งจะลบไฟล์เก่าทิ้งจริงเสมอ จึงไม่มีไฟล์ขยะสะสม)
- เกณฑ์ "20% ของพิกเซลต้องโปร่งใส" เป็นค่าที่กำหนดขึ้นโดยประมาณตามหลักเหตุผล ยังไม่ได้ทดสอบ
  กับตัวอย่างลายเซ็นจริงที่หลากหลาย (เช่น ลายเซ็นที่มีเส้นหนามาก หรือครอบคลุมพื้นที่เยอะผิดปกติ
  อาจต้องพิจารณาปรับเกณฑ์นี้ลงในอนาคตถ้าพบว่าลายเซ็นจริงบางแบบผ่านเกณฑ์ยาก)
